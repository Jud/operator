# MPSGraph Decoder Implementation Notes

Living document of learnings from building the Qwen 3.5 2B decoder in MPSGraph, verified layer-by-layer against PyTorch reference activations.

## Architecture Overview

Qwen 3.5 2B: 24 layers in a 3:1 pattern `[DeltaNet, DeltaNet, DeltaNet, FullAttention]` × 6.
- 18 DeltaNet (linear attention) layers
- 6 full attention layers
- Hidden size: 2048, 16 heads, head_dim: 128, KV heads: 2
- QKV dim: 6144 (3 × 2048), no bias on projections
- FFN intermediate: 6144 (gate + up projections)
- Vocab: 248,320
- Layer indices: DeltaNet = [0,1,2, 4,5,6, 8,9,10, 12,13,14, 16,17,18, 20,21,22], Attention = [3, 7, 11, 15, 19, 23]

## Verification Strategy

**FP32 first, FP16 second.** All ops built and verified in FP32 against PyTorch references. Any nonzero error in FP32 means a real implementation bug. FP16 conversion is a separate step after FP32 verification passes, so FP16 rounding is the only error source and can be measured precisely.

Reference data: `models/reference_activations/` — 528 activation tensors + all weights + cache states, ALL saved in FP32. Generated by `scripts/pipeline/dump_reference.py` which runs one prefill + one decode step through the unmodified PyTorch model with submodule hooks.

## Verified Ops (FP32)

### Embedding Lookup
- **Op**: `gatherND(embedWeight, tokenId)`
- **Gotcha**: Token IDs are int32. PyTorch hooks capture them as float16 which overflows for vocab > 65504 (token 248068 → inf in FP16). Use `decode_inputs/token_id.npy` (int32) not `activations/embed_in_0.npy`.
- **FP32 verification**: maxDiff = 0.000000 (exact match)

### RMSNorm (Qwen3_5RMSNorm — input/post-attention layernorms)
- **Op**: `x * rsqrt(mean(x², dim=-1) + eps) * (1 + weight)`
- **CRITICAL: Qwen uses `(1 + weight)` not just `weight`.** The weight parameter is a residual — initialized to zeros, so `1 + weight` starts at 1.0. This is different from standard LLaMA RMSNorm which uses `weight` directly. Missing the `+1` gives outputs ~25x too small.
- **Source**: `Qwen3_5RMSNorm.forward()` — `output * (1.0 + self.weight.float())`
- **Precision**: Must compute in FP32 throughout, cast to output dtype only at the end.
- **eps**: 1e-6 (from `config.text_config.rms_norm_eps`)
- **FP32 verification**: maxDiff = 0.000000 (exact match)

### RMSNormGated (Qwen3_5RMSNormGated — DeltaNet internal norm)
- **Op**: `weight * (x * rsqrt(mean(x², dim=-1) + eps)) * silu(gate)`
- **DIFFERENT from RMSNorm**: Uses `weight` directly (NO `1 + weight`). Normalizes BEFORE gating. The gate input is `z` (separate projection output).
- **Norm amplification**: Recurrence output is very small (range [-0.035, 0.064]). The norm amplifies by rsqrt(variance) ≈ 24x, then the weight and silu(z) gate bring it to the final range ([-2.4, 2.0]).
- **Source**: `Qwen3_5RMSNormGated.forward()` — `self.weight * hidden_states * F.silu(gate)`

### Linear Projection (no bias)
- **Op**: `reshape(x, [1, K]) → matmul(x, W.T) → reshape([1, 1, N])`
- **Note**: All DeltaNet projections are bias-free: `in_proj_qkv` (6144×2048), `in_proj_z` (2048×2048), `in_proj_a` (16×2048), `in_proj_b` (16×2048), `out_proj` (2048×2048). Conv1d also has no bias.
- **Weight shape**: `[out_features, in_features]` — need transpose for matmul.
- **FP32 verification**: maxDiff = 0.000002 (QKV), 0.000000 (Z), 0.000001 (B), 0.000002 (A)

## DeltaNet Layer (decode step)

The decode step processes one token with precomputed conv/recurrent states.

### Projections (all verified in FP32)
1. `qkv = in_proj_qkv(hidden) → [1, 1, 6144]` — then transposed to `[1, 6144, 1]` for conv
2. `z = in_proj_z(hidden) → [1, 1, 2048]` — reshaped to `[16, 128]` for gated norm
3. `b = in_proj_b(hidden) → [1, 1, 16]` — beta gate input
4. `a = in_proj_a(hidden) → [1, 1, 16]` — alpha/decay gate input

### Conv1d Update (causal_conv1d_update)
- For decode (single token), this is just a dot product per channel:
  1. Shift conv_state left by 1, append new QKV column: `new_state = cat(state[:,:,1:], qkv_t, dim=2)`
  2. Element-wise multiply: `new_state * conv_weight` (both `[1, 6144, 4]`)
  3. Reduce sum along last dim: → `[1, 6144, 1]`
  4. Apply SiLU activation: `x * sigmoid(x)`
- `conv_state = [1, 6144, 4]` (kernel_size=4)
- **Gotcha**: `causal_conv1d_update` returns a tuple `(output, new_state)`, not just the output tensor. Must unpack.
- **Conv1d has no bias** in this model.
- **Verified in Python**: Manual shift+multiply+reduce matches PyTorch exactly (diff 0.0)

### QKV Split
- Split `[1, 1, 6144]` into query `[1, 1, 2048]`, key `[1, 1, 2048]`, value `[1, 1, 2048]`
- Reshape each to `[1, 1, 16, 128]` (16 heads, 128 head_dim)

### L2 Normalization (query and key)
- **Op**: `F.normalize(x, p=2, dim=-1, eps=1e-6)` = `x / max(||x||₂, eps)`
- **CRITICAL**: PyTorch's `F.normalize` uses `x / max(norm, eps)` — it CLAMPS the norm to at least eps. This is DIFFERENT from `x / sqrt(sum(x²) + eps)` which adds eps inside the sqrt. For near-zero vectors, these give wildly different results (1000x difference).
- `use_qk_l2norm_in_kernel=True` means the recurrence function normalizes Q and K internally.

### Gates
- `beta = sigmoid(b)` → `[1, 1, 16]`
- `g = -exp(A_log) * softplus(a + dt_bias)` → `[1, 1, 16]` (decay factor)
- Both computed in FP32. The `A_log` and `dt_bias` are per-head parameters.
- `exp(g)` values range from 0.04 to 1.0 — some heads decay heavily, others barely.

### Recurrent State Update (single step)
```
# All in FP32, shapes: state [16, 128, 128], k/v/q [16, 128], beta [16, 1]
scale = 1 / sqrt(128)
q = q * scale
state = state * exp(g)                           # exponential decay
kv_mem = einsum('hkv,hk->hv', state, k)          # state @ k per head
delta = (v - kv_mem) * beta                       # gated update
state = state + einsum('hk,hv->hkv', k, delta)   # outer product update
output = einsum('hkv,hk->hv', state, q)          # query @ state per head
```
- State shape: `[1, 16, 128, 128]` (batch, heads, key_dim, value_dim)
- The `einsum` operations can be expressed as batched matmul in MPSGraph:
  - `state.T @ k` for kv_mem
  - `k @ delta.T` for outer product
  - `state.T @ q` for output

### Norm + Z Gating (Qwen3_5RMSNormGated)
- Input: recurrence output `[16, 128]` + z projection `[16, 128]`
- `norm(x, z)`: RMSNorm WITHOUT `+1`, then multiply by `silu(z)`
- Output: `[1, 1, 2048]` after reshape

### Out Projection
- `out_proj(normed) → [1, 1, 2048]` — standard bias-free linear

## Performance Notes

- Layer 0 (embed + RMSNorm + QKV matmul): 0.42ms via MPSGraph
- Single [1,2048]×[6144,2048] FP16 matmul: 0.24ms via MPSGraph
- Same matmul with uint8 256-entry LUT dequant: 0.48ms (2x overhead)
- **Same matmul with uint4 16-entry LUT dequant: 0.51ms (2.3x overhead)** ← THE KEY FINDING
- CoreML FP16 full model decode: 25.9ms (24 layers)
- CoreML INT4 bs=8 full model decode: 12ms/tok (84 tok/s)
- CoreML PAL4 gs=16 full model decode: 33ms/tok (not fused — the problem we're solving)

### MPSGraph uint4 LUT — The Breakthrough

**MPSGraph supports `MPSDataType.uInt4` with `dequantize(_:LUTTensor:)`**. 4-bit packed indices (2 values per byte) with a 16-entry FP16 lookup table. Native fusion with matmul.

| Approach | On-disk | Runtime | Speed/layer | Quality |
|----------|---------|---------|------------|---------|
| CoreML PAL4 gs=16 | 900 MB | 900 MB | ~1.4ms (not fused) | 15/15 |
| CoreML INT4 bs=8 | 1.3 GB | 1.3 GB | ~0.5ms (fused) | 15/15 |
| **MPSGraph uint4 LUT** | **~1 GB** | **~1 GB** | **0.51ms (fused)** | **PAL4 quality** |

This is the trifecta: PAL4 quality + INT4 speed + PAL4 size. The uint4 data type was always available in MPSGraph — we just never tested it because the uint8 path was the first thing we tried.

**Weight packing**: uint4 stores 2 values per byte. A [6144, 2048] weight = 6,291,456 4-bit values = 3,145,728 bytes = 3 MB. Full model ~2.1B params → ~1 GB on disk.

**LUT per weight tensor**: Each weight matrix gets its own 16-entry FP16 LUT (32 bytes). For grouped palettization quality, we'd need per-group LUTs — but MPSGraph's `dequantize(_:LUTTensor:)` with a rank-1 LUT is per-tensor only. Per-group would need the `axis:` variant, which we need to test next.

## Hypotheses Tested

| # | Hypothesis | Result | Notes |
|---|-----------|--------|-------|
| 1 | FP16 references are sufficient for verification | **FAILED** | FP32→FP16 rounding compounds through DeltaNet recurrence. maxDiff 0.57 from FP16 inputs. Must use FP32 references. |
| 2 | Monkey-patching forward captures clean references | **FAILED** | Changes prefill behavior for all 18 DeltaNet layers, corrupts all decode references. Use submodule hooks only. |
| 3 | Qwen RMSNorm uses standard `x * weight` | **FAILED** | `Qwen3_5RMSNorm` uses `x * (1 + weight)`. Missing +1 gives 25x error. |
| 4 | Submodule hooks capture correct decode activations | **CONFIRMED** | Hooks fire on unmodified model. All 6 projections match FP32 exactly. |
| 5 | MPSGraph matmul matches PyTorch for FP32 linear ops | **CONFIRMED** | maxDiff 0.000002 — pure floating point rounding. |
| 6 | Conv1d update = dot product for single-token decode | **CONFIRMED** | Manual shift+multiply+reduce matches PyTorch exactly (diff 0.0). |
| 7 | F.normalize uses `sqrt(sum + eps)` | **FAILED** | Uses `max(||x||, eps)` — clamp, not add-inside-sqrt. |
| 8 | DeltaNet black box produces correct output range | **CONFIRMED** | Output range [-2.39, 1.99] matches reference [-2.40, 2.02]. |
| 9 | Build FP32 first, convert to FP16 after | **ADOPTED** | FP32 verification shows maxDiff < 0.000002 across all ops. Zero implementation bugs. FP16 conversion measured separately. |
| 10 | Qwen3_5RMSNormGated uses same formula as RMSNorm | **FAILED** | Gated variant uses `weight * normalized * silu(gate)` — NO `1+weight`. Different class, different formula. |
| 11 | MPSGraph requires uint8 for LUT dequant | **FAILED** | `MPSDataType.uInt4` exists and works with `dequantize(_:LUTTensor:)`. 4-bit packed (2 values/byte), 16-entry LUT. 0.51ms per realistic layer. Model stays ~1 GB. |

## Verification Architecture

```
PyTorch (ground truth, FP32)
    ↓ dump_reference.py (hooks capture all activations + weights as FP32 .npy)
Reference files (models/reference_activations/)
    ↓ VerifyFP32.swift (MPSGraph ops in FP32, compare against reference)
FP32 verified graph
    ↓ (future) convert to FP16, measure rounding error
    ↓ (future) add LUT dequant, measure quantization error
Production graph
```

Key principle: **each conversion step adds one measurable error source.** No ambiguity about where errors come from.

### Reference dump script
- `scripts/pipeline/dump_reference.py`
- Loads model in FP32, runs prefill + one decode step
- Hooks on all submodules capture input/output for every projection, norm, FFN
- Saves activations, weights, cache states, RoPE, token IDs, expected logits
- **No monkey-patching** — hooks only, on the unmodified model
- Clears activation dict between prefill and decode to capture only decode step

### Swift verification
- `Operator/Sources/MPSGraphDecoder/VerifyFP32.swift` — FP32 ops against FP32 references
- `NpyLoader` — loads .npy files (FP16, FP32, int32) into MTLBuffers
- `compareBuffers` — reports maxDiff, avgDiff with configurable tolerance and FP16/FP32 mode

- **Conv1d → Conv2d**: ANE prefers Conv2d. Could reshape for ANE acceleration. Only matters for prefill.
- **ANE scheduling**: Explore ANE compute unit targeting for specific ops.

## General Learnings

- **FP32 first**: Build and verify all ops in FP32 before touching FP16. Any FP32 error is a real bug.
- **Never monkey-patch**: Submodule hooks capture clean references. Monkey-patching forward methods changes model behavior and corrupts references across all layers.
- **Two Qwen norms**: `Qwen3_5RMSNorm` (1+weight, for input/post-attention) and `Qwen3_5RMSNormGated` (weight only, for DeltaNet internal). Don't confuse them.
- **L2 norm = clamp, not add**: `F.normalize` uses `max(||x||, eps)`, NOT `sqrt(||x||² + eps)`.
- **Token IDs overflow FP16**: Token 248068 > 65504. Always use int32 for token IDs.
- **Save references in FP32**: FP16 rounding compounds through the DeltaNet recurrence (outer product on [16,128,128] state).
- **SPM caching**: `swift build` sometimes doesn't rebuild after edits. Use `swift package clean` to force.
- **MPSGraph LUT constraints**: `dequantize(_:LUTTensor:)` requires power-of-2 LUT size = 2^(index_bits). uint4 → 16 entries, uint8 → 256 entries. **uint4 is supported and fast** — use it for PAL4-quality weights at ~1 GB.
- **Always check the actual API**: We assumed MPSGraph required uint8 because the first test used uint8. Checking `MPSDataType.uInt4` would have saved hours of workaround exploration.
- **MPSGraph buffer size**: FP32 embedding weight (248K × 2048 × 4 bytes = 2GB) needs large MTLBuffer support.
- **CoreML PAL4 fusion**: Only per_tensor 16-entry LUT fuses with linear. Grouped LUTs (384×16) and larger LUTs (256) do NOT trigger the fast kernel path.
